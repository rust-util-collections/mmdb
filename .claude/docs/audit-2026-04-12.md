### [CRITICAL] manifest: Trivial-move compaction file lost on MANIFEST recovery

**Where**: `src/manifest/version_set.rs:145-156`
**What**: MANIFEST recovery uses a `HashMap<u64, (usize, FileMetaData)>` keyed by file number. Within each edit, additions (`new_files`) are processed before deletions (`deleted_files`). A trivial-move edit contains both `add_file(target_level, meta)` and `delete_file(source_level, meta.number)` for the **same** file number. The HashMap insert succeeds, then the immediately following deletion removes it — the file disappears entirely from the live set.
**Why**: `log_and_apply` (line 278-283) scopes deletions to a specific level, so the target-level copy survives at runtime. But `recover` deletes from the flat HashMap by file number alone, losing the level distinction. Matches pattern 1.3 (Compaction + Flush Race variant — data silently disappears).
**Evidence**: Trivial move at `src/compaction/leveled.rs:664-669` produces an edit where `deleted_files` and `new_files` share the same file number. Recovery at line 149 inserts, then line 155 removes the same key.
**Fix**: Process `deleted_files` **before** `new_files` within each edit during recovery:
```rust
// Deletions first, then additions
for (_level, file_number) in &edit.deleted_files {
    live_files.remove(file_number);
}
for (level, meta) in &edit.new_files {
    live_files.insert(meta.number, (*level as usize, meta.clone()));
}
```

### [MEDIUM] SST: Zstd decompression has no size bound check

**Where**: `src/sst/table_reader.rs:444-446`
**What**: The LZ4 decompression path validates declared size against `MAX_DECOMPRESSED_BLOCK_SIZE` (64 MB) before allocating. The Zstd path calls `zstd::stream::decode_all()` with **no size bound**. A crafted SST with a Zstd block decompressing to an arbitrarily large buffer causes unbounded allocation.
**Why**: SST pattern guide: "Decompressed block size bounded to prevent allocation bomb." LZ4 defends against this; Zstd omission is inconsistent.
**Fix**: Use `zstd::bulk::decompress(&data, MAX_DECOMPRESSED_BLOCK_SIZE)` with an explicit capacity limit.

### [MEDIUM] iterator: iterate_lower_bound not enforced during backward iteration

**Where**: `src/iterator/db_iter.rs:591-728` (`resolve_prev_user_key`)
**What**: `iterate_lower_bound` is stored and can be set via `set_lower_bound`/`set_bounds`, but `resolve_prev_user_key` never checks it. Backward iteration (`prev()`, `seek_to_last()`, `seek_for_prev()`) produces entries with user keys below the lower bound. The lower bound is propagated to sub-iterators via `set_bounds`, but all sub-iterators discard it: `TableIterator::set_bounds` ignores `_lower`, `LevelIterator::set_bounds` ignores `_lower`, and `MemTableCursorIter` uses the default no-op.
**Why**: Violates the API contract of `set_lower_bound` (documented as "inclusive lower bound on user keys"). Affects `iter_with_range()` callers who set both bounds and iterate backward.
**Fix**: Add a lower-bound guard in `resolve_prev_user_key` after extracting the user key:
```rust
if let Some(ref lb) = self.iterate_lower_bound
    && uk < lb.as_slice()
{
    self.current = None;
    self.needs_advance = false;
    self.backward_positioned = false;
    return;
}
```

### [LOW] compaction: Orphan SST files on merger error (2 locations)

**Where**: `src/compaction/leveled.rs:457-476` and `src/compaction/leveled.rs:1114-1134`
**What**: When `merger.error()` returns `Some(e)`, the function returns `Err` but SST files already written during the merge loop are not cleaned up. The `new_files` vector is dropped without the caller seeing it, leaving orphan files on disk.
**Why**: Partial failure should clean up intermediate artifacts. Files not referenced by MANIFEST are orphaned until recovery.
**Fix**: Before returning the error, iterate `new_files` and delete SST files from disk.

### [LOW] compaction: force_merge_level does not invalidate block cache

**Where**: `src/compaction/leveled.rs:1136-1148`
**What**: `force_merge_level` evicts old input files from the table cache but never calls `block_cache.invalidate_file()`. Compare with `install_compaction` (lines 890-898) which correctly invalidates both caches.
**Why**: Stale data blocks from deleted SST files waste cache capacity until LRU eviction. Not a correctness bug (file numbers are never reused).
**Fix**: Accept `block_cache: Option<&Arc<BlockCache>>` and call `bc.invalidate_file(*num)` alongside table cache eviction.

### [LOW] iterator: peek_source_level returns usize::MAX before heap initialization

**Where**: `src/iterator/merge.rs:1208-1216`, called from `src/iterator/db_iter.rs:322`
**What**: In `next_visible()`, `peek_source_level()` is called before `peek_entry()` which triggers `init_heap()`. On the first call when the heap is uninitialized, `peek_source_level` returns `usize::MAX`. This maps to `level_filter = None`, so all tombstones are checked regardless of level — conservative direction (may over-delete, never under-delete).
**Why**: Pattern 2.5 (range tombstone boundary). The first entry pays a tombstone check penalty but no data correctness risk.
**Fix**: Call `peek_source_level()` after `peek_entry()` to ensure the heap is initialized.

### [LOW] iterator: LevelIterator seek_for_prev with prefix filter may miss entries beyond 2-file window

**Where**: `src/iterator/level_iter.rs:218-255`
**What**: `seek_for_prev` limits its search to at most 2 candidate files. If both are filtered out by prefix bloom, files at lower indices that may contain keys <= target are not checked.
**Why**: Invariant INV-I2 (no missing keys) — backward seek could miss the largest key <= target when it exists in a file outside the 2-file window.
**Fix**: When both candidates are filtered, continue scanning backward through earlier files until one passes or all are exhausted.

### [LOW] memtable: Size tracking does not account for range tombstone duplicate storage

**Where**: `src/memtable/mod.rs:52-68`
**What**: `RangeDeletion` entries are stored in both the skiplist and the `range_tombstones` Vec, but `entry_size` only accounts for the skiplist entry. The duplicated data in `range_tombstones` is untracked.
**Why**: Violates INV-M4 (Size Tracking Accuracy). Minor in practice since range tombstones are rare.
**Fix**: Add `tombstone_overhead = key.len() + value.len() + size_of::<MemRangeTombstone>()` to the size calculation for `RangeDeletion` entries.

### [LOW] WAL: Fragmented record truncation at EOF returns hard error

**Where**: `src/wal/reader.rs:71-76`
**What**: When EOF is reached mid-fragmented-record, `read_record()` returns `Err(Corruption("partial record without end"))` instead of `Ok(None)`. The caller in `db.rs:328-331` handles this as non-fatal (warning + break), so no data loss occurs.
**Why**: INV-W4 states incomplete records at EOF should be silently skipped. The reader's contract is stricter than specified.
**Fix**: Return `Ok(None)` instead of `Err(Corruption(...))` when EOF with `in_fragmented_record`.

### [LOW] rate_limiter: Sleep duration based on total accumulated debt

**Where**: `src/rate_limiter.rs:59-63`
**What**: When `available` is already negative from prior callers, a new caller deducts its `needed` then sleeps for `(-total_available) / rate` — the full debt, not just its own shortfall. This means O(N^2) total sleep time across N concurrent callers and effective throughput below the configured rate.
**Why**: Rate limiter correctness. Each caller should sleep only for its own deficit.
**Fix**: Compute sleep from the caller's own shortfall: capture `pre_deduct = available`, deduct, then sleep for `(needed - pre_deduct.max(0.0)) / rate`.
